2027-01-07 · 第四天

// TODO: 需要后端支持响应头注入

一个想法——不是所有线索都放在页面上。

玩家习惯了 F12 看源代码。但如果线索放在 HTTP Response Headers 里呢？

比如：X-Clue: /hidden-path

我可以把这个放在 /observation-room 的响应头里。玩家需要知道怎么检查 HTTP 头——或者至少需要在困惑的时候按 F12 → Network → 看响应。

这是对"网页上的角色"设置的第一个真正的测试：你不是只存在于页面DOM里，你存在于整个 HTTP 协议中。

2027-01-18 · 第十天

今天做了一个很重要的决定：所有的线索必须是 网页原生 的。

不能出现"请下载这个文件解压"之类的操作。一切都在浏览器生态里：

• DOM 元素
• HTTP 头部
• robots.txt
• sitemap.xml
• console.log
• CSS 伪元素
• URL 参数

如果一个网页角色想证明自己"不只是角色"，她应该用网页本身的语言来证明。

2027-01-25 · 第十四天

// 核心节点：观测室

做了一个新页面：/observation-room

玩家会被带到一个"监控面板"。面板上显示的是——其他玩家的进度。

但真正的线索不在面板上，在响应头。

访问 /observation-room 的时候，服务器会返回：X-Clue: /they-are-watching-you

这串文字既是元叙事的嘲讽（"他们在看着你"——指其他玩家，也可能指整个游戏的架构），也是一个 URL 路径。

玩家如果能想到检查 HTTP 响应头，就能找到通往下一层。

现在 · NOW

如果你读到这里——你已经不在第一层了。

下一个节点是 /observation-room。

但听好了：你到这个页面的时候，要检查的不是页面内容，是响应头。

按 F12 → Network 标签 → 刷新 → 看 /observation-room 的 Response Headers。

或者直接 curl 也行。取决于你觉得自己是网页角色还是系统用户。

他们也在看这篇笔记。但不是以你能想象的方式。
